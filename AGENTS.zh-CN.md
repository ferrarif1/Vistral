# Vistral Codex 协作指南（中文版）

## 1）项目使命与范围
Vistral 是一个**AI-native、对话与附件驱动**的视觉模型平台。

本仓库当前阶段目标：
- 将产品蓝图（PRD / IA / Flows / Data Model / API Contract）落地为可持续演进的工程基线；
- 保持两类系统角色（User、Admin）与资源所有权权限模型的一致语义；
- 支持模型训练、审批、发布、边缘部署、审计等核心能力。

**范围约束**：本文件定义的是 Codex 在本仓库内的协作规则，不是产品运行时“内部 agent”设计文档。

## 2）开工前必读顺序（必须遵守）
1. `README.md`（仓库定位与入口）
2. `AGENTS.md`（协作规则）
3. `.codex/config.toml`（执行与安全边界）
4. `docs/prd.md`（需求与非功能要求）
5. `docs/ia.md`（信息架构）
6. `docs/flows.md`（关键流程）
7. `docs/data-model.md`（数据模型与状态枚举）
8. `docs/api-contract.md`（接口契约）
9. `.agents/skills/**/SKILL.md`（按任务调用）

若文档冲突，按“更具体、更新近、影响面更大”优先，并在变更说明中记录。

## 3）Skill 调用规则
- 产品规划 / 方案拆解：`product-architect`
- 前端一致性审查：`frontend-parity-review`
- 后端合同与状态机审查：`backend-contract-guard`

多 skill 同时命中时建议顺序：
1) `product-architect` → 2) 前/后端守护 skill → 3) 统一回归检查。

## 4）全站强约束
1. 不能只做首页：新增能力必须覆盖完整入口与主流程；
2. 上传附件必须可见 / 可删 / 有状态（uploading / processing / ready / error）；
3. 多步骤流程必须有顶部 stepper；
4. 高级参数默认折叠；
5. 空态 / 错态 / 加载态 / 成功态统一。

## 5）文档优先
涉及行为、数据、接口、流程改动时，先更新合同文档，再写实现：
- 流程改动先改 `docs/flows.md`
- 数据改动先改 `docs/data-model.md`
- 接口改动先改 `docs/api-contract.md`
- 需求改动先改 `docs/prd.md`（必要时同步 `docs/ia.md`）

## 6）改动原则
- 优先共享层与可复用组件；
- 不允许页面风格和交互规则漂移；
- 相同语义能力（上传列表、stepper、状态反馈）应抽象复用；
- 保持 AI-native 体验语义一致。

## 7）验证要求
每次改动后至少执行与改动面匹配的最小检查：
- 文档改动：链接有效性 + 术语一致性；
- 前端改动：lint/typecheck + 关键状态自测；
- 后端改动：lint/test + 合同对齐检查。

受环境限制无法完成时，需在交付说明中明确未执行项与风险。

## 8）输出要求
每次任务都遵循：
1. 先 plan；
2. 再改动（先文档合同，后实现）；
3. 最后风险自查。

交付说明至少包含：改了什么、为什么改、验证了什么、剩余风险与下一步建议。

## 9）中断交接记录（强制）
如果“新对话任务会打断当前未完成工作”，必须先完成交接记录，再切换任务。

最小动作：
1. 在 `docs/work-handoff.md` 追加一条交接记录（不能覆盖历史）；
2. 记录已完成项、未完成项、下一步执行顺序、主要风险/阻塞、建议验证命令；
3. 若里程碑顺序变化，必须同步更新 `PLANS.md`；
4. 对用户回复中明确“已记录交接信息”并给出文档路径。

记录字段至少包含：
- `date_time`
- `context`（当前目标/被什么任务打断）
- `done`
- `next`
- `risks`
- `verification`
