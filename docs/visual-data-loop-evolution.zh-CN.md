# 视觉数据闭环演进方案（借鉴 Roboflow，保持 Vistral 原生定位）

## 1. 背景与目标
Vistral 需要吸收成熟视觉数据产品（如 Roboflow）在数据组织与操作效率上的优点，但不做页面克隆。

目标结果：
- 保持 Vistral 的 `chat-first` 入口与对话编排能力
- 强化专业工作台在视觉数据闭环中的执行效率
- 提升闭环质量：上传 -> 标注 -> 复核 -> 版本 -> 训练 -> 推理验证 -> 错样回流

本文件作为后续多轮实现的执行基线。

## 2. A）最值得迁移到 Vistral 的机制（按优先级）

### P0（优先落地）
1. 数据集样本浏览器（从“列表展示”升级为“可操作工作面”）。
   - 图片网格/列表切换
   - 快速筛选（搜索、split、状态、队列状态、class 命中）
   - 批量选择 + 批量操作（面向样本）
2. 单样本复核工作台。
   - 在同一屏中查看原图、标注、复核上下文、元数据与操作动作
   - 目标是“高吞吐复核与修正”，不是普通详情页
3. 版本中心化表达。
   - dataset version 提升为一等对象
   - 训练/导出/评估入口显式绑定版本快照
4. 预测与标注对比入口。
   - overlay 控件 + 置信度导向的排查提示
   - 更快发现漏检/误检/低置信样本

### P1（P0 之后高价值）
1. tag / metadata 运营化。
   - 不只展示，必须可筛选、可批量更新
   - 参与 triage 分流与训练范围准备
2. 队列化复核体验强化。
   - `needs_work / in_review / rejected / approved` 作为一等流程泳道
   - 背景刷新下仍保持稳定队列推进与快捷键体验
   - 队列筛选状态以标签持续可见并可一键清空，同时提供 metadata 快捷筛选预设用于高频排查

### P2（核心闭环稳定后）
1. 常用筛选视图（保存过滤器）
2. 错误模式切片（面向主动学习）
3. 跨版本结果对比与数据血缘可视化增强

## 3. B）结合当前仓库，优先改哪些页面/组件/对象

## 3.1 优先页面
1. `/datasets/:datasetId`
   - 增强样本浏览区：筛选 + 批量选择 + 批量动作 + 样本卡片
   - 强化版本面板：作为训练/导出入口枢纽
2. `/datasets/:datasetId/annotate`
   - 收口为“样本复核台”（图像/标注/复核信息/元数据同屏）
3. `/inference/validate`
   - 强化“推理结果回流复核”入口，并保持任务类型守卫
4. `/training/jobs/new`
   - 强化“版本快照可复现”可见性与启动前确认

## 3.2 共享组件（新增或抽象）
1. `DatasetItemBrowser`（网格/列表、筛选条、批量选择）
2. `SampleReviewWorkbench`（单样本复核工作面）
3. `DatasetVersionRail` / `VersionSnapshotCard`
4. `PredictionOverlayControls`（标注/预测/置信度显示控制）
5. `BulkActionBar`（批量 split/status/tag 更新）

全部复用既有 UI 基础组件：`Card`、`Panel`、`Badge`、`StatusTag`、`Workspace*`、`StateBlock`。

## 3.3 数据对象演进（增量、契约对齐）
1. Dataset item metadata 约定化：
   - 预留可运营键（如 `tag:*`、`source`、`scenario`、`difficulty`）
2. 复核上下文：
   - 继续以样本粒度暴露 latest review reason/comment
3. 预测回流元数据：
   - 持续保留 run/source/reason 可追溯链路

坚持增量演进，不推倒现有实体模型。

## 4. C）分阶段实施方案（先后顺序与原因）

### 阶段 1（最小可用结构升级）
1. Dataset detail：
   - 增加样本浏览器筛选 + 网格/列表 + 批量选择
   - 基于现有 item patch 接口实现批量 split/status 更新
2. Annotation workspace：
   - 收敛样本复核台信息布局
   - 提升 latest review context 与队列推进可见性
3. 文档对齐：
   - 在 PRD/IA/Flows 显式写入浏览器/复核台/版本中心化路径

原因：
- 对操作效率提升最大，且后端风险低
- 主要是结构与交互层改造，可复用现有 API

### 阶段 2（版本与对比能力增强）
1. 版本面板增强：
   - 版本就绪度 + 下游动作（训练/导出/对比）直达
2. 预测对比：
   - overlay 开关、置信度阈值、低置信样本提示
3. 闭环串联：
   - 推理反馈回到标注队列的路径更清晰

原因：
- 让可复现训练和验证闭环在产品层面真正显性化

### 阶段 3（运营智能增强）
1. 保存筛选视图
   - 第一版已落地：数据集样本浏览器支持本地视图预设的保存 / 应用 / 删除
   - 样本浏览器现已显示“当前生效筛选条件”并支持一键清空，减少长时间整理时的筛选漂移
   - 错误模式切片与 metadata/tag 切片卡片已支持「保存为视图」，可把高价值排查切片直接沉淀为复用视图
2. metadata/tag 驱动采样策略
   - 第一版已落地：数据集详情侧栏新增 metadata/tag 信号切片（高频 tag、source 值、feedback_reason），支持一键聚焦样本浏览器并跳转对应队列
   - 数据集样本浏览器与标注队列的 metadata 过滤都已支持 `key=value` 查询语法
3. 错误模式切片
   - 第一版已落地：数据集详情侧栏新增常见问题切片（拒绝原因聚类、低置信标签、推理回流样本、已就绪但未分配 split），支持一键聚焦
4. 版本间结果对比摘要
   - 第一版已落地：`DatasetVersionRail` 支持“当前激活版本 vs 上一个版本”的样本数、覆盖率、split 变化摘要

原因：
- 在核心闭环稳定后，进一步提升团队长期运营效率

## 5. 执行护栏
1. 不做 Roboflow 克隆：
   - 保留 Vistral chat-first 产品定位与 IA
2. 一套对象语言：
   - 对话工作台与专业工作台共享同一语义与状态词汇
3. 优先结构效率，不做功能堆砌：
   - P0/P1 未稳定前，避免大规模后端重构
4. 文档合同优先：
   - 行为变更前后必须同步合同文档
