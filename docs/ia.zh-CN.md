# 信息架构（中文版）

## 1. 目标
定义 Vistral 可执行路由与页面结构，覆盖 AI-native 对话工作区与专业工程控制台双入口。

## 2. 访问规则
- 系统角色仅 `user` 与 `admin`。
- `owner` 是资源关系，不是系统角色。
- 当前关闭公开自助注册。
- 账户开通由管理员在已登录的账户设置界面内完成。

## 3. 路由地图

### 3.1 认证
- `/auth/login`
- `/auth/register`
  - 重定向到 `/auth/login`
  - 不再提供公开自助注册界面

### 3.2 双入口
- `/`
  - AI-native 对话工作区入口
  - 专业控制台入口

### 3.3 对话工作区
- `/workspace/chat`
  - 沉浸式 chat 壳层（左侧会话栏 + 中央时间线 + 底部浮动输入区）
  - 顶部页头保持轻量，主要保留模型与模式信息；语言切换统一收敛到“点击用户头像/账号卡片”弹出的账户菜单中
  - 模型选择器仅展示整理后的基座/基础模型目录，用于后续训练与日常对话入口
  - 左侧会话栏接入后端会话历史，支持同步与点击恢复完整对话详情
  - 桌面端左侧栏保持固定视口高度，使用内部滚动而不是跟随整页无限拉长
  - 桌面端侧栏支持折叠/展开，可在“专注对话”和“浏览历史”之间切换
  - 移动端侧栏采用临时抽屉形态，可从页头打开，并通过遮罩或关闭动作收起
  - 左侧会话历史默认直接列表展示（不提供搜索框，不展示快捷入口区块，不显示“最近会话”等分组标题）
  - 当历史为空时，列表区保持空白，不额外渲染占位标题文案
  - 置顶会话仍支持拖拽排序
  - 桌面端会话项默认保持简洁，只在鼠标移入时显示三点溢出按钮，避免动作按钮长期占位
  - 会话项支持溢出/快捷菜单（桌面右键、点击三点，或移动端长按）；打开会话保留为点击整行，菜单内提供重命名/置顶/删除
  - 菜单支持键盘导航（`ArrowUp/ArrowDown`、`Enter`、`Esc`）与快捷键（`R/P/D`）
  - 移动端长按在浏览器支持振动 API 时触发轻量震动反馈
  - 左侧底部账户区保持常显且不可折叠，以“头像 + 用户名 + 身份（普通用户/管理员）”作为入口
  - 输入区内按需展开的附件托盘
  - 当前草稿附件以标签形式显示，发送后自动从输入区收起
  - 附件托盘的后台刷新必须保持安静，只在附件数据真实变化时更新可见草稿/文件状态
  - 助手消息可渲染真实后台操作卡片（`create_dataset` / `create_model_draft` / `create_training_job`）
  - 当操作参数不完整时，对话在线程内继续补齐，仅追问缺失字段

### 3.4 专业控制台
- `/workspace/console`
  - 采用共享应用壳层（全局左侧导航 + 顶部上下文栏），不再复用对话页沉浸式布局
  - 所有工程控制台页面都必须遵守“单页单主任务”原则：
    - 一页只承载一个核心操作任务
    - 其他任务域只能以摘要信息或跳转入口出现
    - 不能因为功能相关就把第二个完整工作流直接嵌进当前页
  - 控制台首页应聚焦“任务分流”：一个优先队列 + 各任务域跳转入口，不再内嵌多个完整流程模块
  - 专业工作页统一使用三段式工作台：
    - 顶部上下文工具栏（`WorkspaceContextBar`）承载搜索/筛选/批量操作
    - 中间主工作区承载资产列表、画布、任务表格等核心内容
    - 右侧检查器面板承载当前选中对象详情与关键后续动作
  - 左侧导航、中间主区、右侧面板在壳层内独立滚动，避免整页混滚
  - 数据集/训练/模型等运营页面优先复用 `WorkspaceWorkbench`，保持结构与交互一致
  - 迭代更新（2026-04）：高频控制台页面已把重量级 onboarding 卡片收敛为紧凑“单主任务提示块”，降低首屏噪音，同时保留明确下一步入口

### 3.5 模型域
- `/models/explore`
- 复用共享 overview 布局：hero 摘要、信号卡、主目录列表、侧边跟进动作
- 页面级新手引导卡需解释如何浏览共享模型目录、识别已就绪与真实性风险模型，并继续进入“我的模型”或版本工作流
- 默认模型目录隐藏 smoke / verify / demo 等内部测试产物；如需演示数据，仅保留少量整理后的样例
- 管理员在模型目录页可直接删除符合条件的非基座模型；受保护的基座/基础模型只显示保护标记，不提供删除入口
- `/models/my-models`
- 复用共享 overview 布局，突出所有权范围状态与下一步动作
- 页面级新手引导卡需解释草稿、待审批、可用模型如何在“我的模型”路径中推进
- `/models/create`（必须 stepper）
  - 页面级新手引导卡需解释 `元数据 -> 工件上传 -> 参数 -> 提交审批`，帮助首次使用者理解模型草稿创建流程
  - 模型文件上传步骤允许后台刷新状态，但文件列表更新必须保持安静，只在文件数据真实变化时生效
- `/models/versions`
- 复用共享 overview 布局，并持续展示版本注册与已完成任务跟进入口
- 页面级新手引导卡需说明如何从已完成训练证据推进到版本注册，并完成版本血缘核查跟进
- 主工作区以及“版本列表为空 / 尚未选择版本”等空态也应同步镜像当前第一个未完成的版本步骤，并提供明确的下一步动作卡片
- 已完成训练详情页可以直接深链到这里，并预填训练任务，便于从完成态开始注册版本；但模型选择仍需显式确认
- 当已完成任务没有任何与其任务类型匹配的自有模型时，注册界面应直接提供带该任务类型预填的模型草稿创建入口，不要静默默认选中不相关模型
- 从已完成任务打开的模型草稿页应持续保留版本注册回流入口，方便带着同一个任务上下文返回

### 3.6 数据集域
- `/datasets`
  - 列表 + 创建入口
  - 页面级新手引导卡需说明这是“数据准备入口”，并引导用户先进入数据集详情页继续流程
  - 引导需展示最小真实进度信号（如 `has_dataset`、`has_ready_dataset`）与“下一步点哪里”
  - 主工作区与空态也应镜像当前第一个未完成的数据准备步骤；当还没有数据集时，应提供直达页内创建面板的动作
- `/datasets/:datasetId`
  - 详情页
  - 支持可选查询参数 `?version=<dataset_version_id>`，用于预选当前活动版本快照
  - 页面级新手引导卡需说明如何从上传/切分/版本推进到标注与训练动作
  - 引导需根据附件状态、标注进展、版本快照等真实数据反映完成度
  - 主工作区也应镜像当前第一个未完成的详情步骤，用一张明确的“下一步卡片”承接用户动作，避免反复通读整页
  - 顶部 stepper（导入/切分/版本）
  - 数据集附件始终可见/可删/有状态
  - 样本浏览区支持网格/列表切换、快速筛选、批量样本操作
  - 筛选项至少覆盖：搜索、split、样本状态、标注队列状态、metadata/tag 提示
- `/datasets/:datasetId/annotate`
  - 样本复核工作台布局（原图 + 标注 + 审核上下文 + 元数据同屏）
  - 支持可选查询参数 `?version=<dataset_version_id>`，用于在复核队列中保留数据集版本上下文
  - 页面级新手引导卡需解释复核流程（`选择 -> 标注 -> 提交/审核 -> 继续队列`）
  - 主工作区与队列空态也应镜像当前第一个未完成的复核步骤，给出明确的下一步动作卡片
  - 引导需保留“返回数据集详情 / 前往验证”直达入口
  - 这页的页内引导默认应保持轻量摘要，完整说明放到持续可见的 `Page tips` 浮动入口中，避免占用主画布工作面
  - 默认页面结构应保持聚焦：
    - 左侧队列栏负责样本选择与筛选
    - 中间主工作面负责预览/画布/编辑器与主要标注动作
    - 右侧检查栏默认只保留当前样本摘要、预测对比、复核动作
  - 队列筛选应分层呈现：
    - 常用筛选（队列、搜索、split）默认可见
    - 低频筛选（样本状态、metadata/tag 表达式）放到折叠的“更多筛选”里
  - OCR 编辑应优先服务最简单动作：
    - 文本输入框与“添加 OCR 行”保持首屏可见
    - 置信度、区域绑定等可选项放到紧凑折叠区中
  - 标注动作应按意图分组，而不是平铺成一整排同级按钮：
    - `继续编辑` 动作（如撤销、保存进行中）
    - `推进流程` 动作（如标记已标注、提交复核）
  - 右侧辅助区应先显示紧凑的复核上下文概览，完整的队列定位、会话历史和雷达明细继续通过折叠区按需展开
  - 当前样本摘要应同时吸收最近一次复核结论（状态 / 原因 / 备注），避免用户在同一样本上来回对照两张卡片
  - 次级上下文（如快捷键总表、复核会话历史、低置信雷达、工作点返回位）默认应折叠或降级展示，避免首次进入时被大量辅助工具淹没
  - 这页用户文案应避免直接暴露生硬的枚举值或状态码，优先使用清晰的中文标签
  - 检测框 + OCR 文本标注
  - 提交审核与通过/拒绝

### 3.7 训练域
- `/training/jobs`
  - 任务列表
  - 支持通过查询参数传入数据集/版本上下文（`?dataset=<id>&version=<id>`），用于从数据集详情版本快照动作直达过滤视图
  - 页面级新手引导卡需解释队列语义（`active` 与 `terminal`）及下一步入口（新建任务、查看详情）
  - 主工作区与空态也应镜像当前第一个未完成的训练控制步骤，并提供清晰的下一步动作
  - 首次进入页面可显示阻塞式加载态，但后台刷新必须保持稳定，不得反复闪回整块加载态；仅在任务数据真实变化时更新可见内容
  - 同时保留手动刷新入口，方便操作人员显式控制
- `/training/jobs/new`
  - 创建向导（必须 stepper，高级参数默认折叠）
  - 页面级新手引导卡需强调“基于可复现版本快照训练”，当就绪条件不满足时提供返回数据准备页的直达入口
  - 主工作区也应镜像当前第一个未完成的训练准备步骤，给出明确的下一步动作卡片；当被门禁阻断时，应明确指向数据集详情或 Runtime 设置
- `/training/jobs/:jobId`
  - 单任务训练运行的状态、日志、指标与产物就绪度详情
  - 调度历史、fallback 原因、技术标识等内容默认后移到高级折叠区
  - 首屏优先围绕“证据查看”（状态/日志/指标/产物），跨域后续操作仅保留轻量跳转入口
  - 详情页需提供直达训练驾驶舱的 `Open cockpit` 入口
  - 已完成任务应提供直达版本注册的入口，并携带训练任务上下文预填到 `/models/versions`
  - 若没有任何自有模型与该任务类型匹配，详情页还应直接提供带任务类型预填的模型草稿创建入口
- `/training/jobs/:jobId/cockpit`
  - 单任务训练驾驶舱
  - 可从训练列表与训练详情页进入，不替代现有详情页的证据审阅职责
  - 采用深色专业控制台布局：顶部摘要、阶段流、指标/资源主面板、自动调参侧栏、底部事件流
  - 支持 `live` / `demo` 模式切换
  - `demo` 模式需提供 `播放 / 暂停 / 重播 / 1x / 2x / 4x`
  - `live` 模式优先显示真实后端数据；缺失的资源/调参流必须明确标记为“不可用/派生”，不能伪装成真实持久化事实

### 3.8 推理验证域
- `/inference/validate`
- 仅展示 runtime 就绪摘要（reachable / unreachable / not configured），完整诊断与配置统一跳转 `/settings/runtime`
- 支持通过查询参数传入数据集/版本上下文（`?dataset=<id>&version=<id>`），用于版本快照场景下的快速验证
- 同时支持 `?modelVersion=<id>` 直接预填模型版本，方便聊天或动作卡片跳转
- 页面级新手引导卡需用通俗语言解释“验证 -> 回流”的闭环关系，并持续显示带上下文的后续入口（数据集/标注）
- 主工作区以及关键空态（`No Model Versions Yet`、`No Ready Inputs Yet`、`No Runs Yet`）也应镜像当前第一个未完成的验证步骤，并给出明确的下一步动作卡片
- 上传图片
- 选择模型版本
- 执行推理
  - 展示 raw + normalized 输出
  - 错样回流数据集（目标数据集 `task_type` 必须与推理任务一致）

### 3.9 设置
- `/settings`
  - 在控制台左侧导航中作为唯一顶层设置入口
  - 页面内通过子选项卡区分：
    - `account`
    - `llm`
    - `runtime`
    - `runtime_templates`
    - `workers`
  - `/settings/account` 作为默认落点，方便优先处理密码与账户管理
  - `/settings/account` 应提供页面级新手引导卡，说明 `确认身份 -> 修改密码 ->（管理员）目录治理 -> 继续 LLM/runtime 配置`
  - 主工作区也应镜像当前第一个未完成的账户设置步骤，并在管理员目录的空态/筛选空态中尽量复用同一条引导
  - 所有已登录用户都可在 `account` 子页修改密码
  - 管理员会在同一 `account` 子页看到开通账户、账号目录、密码重置，以及停用/恢复账号动作
  - 在账号目录中停用账号前，需先以内联表单填写原因；恢复账号则保持轻量单击动作
  - 账号行至少展示角色、状态、创建时间、最近登录时间，以及在停用状态下展示已记录的停用原因
  - `/settings/llm`、`/settings/runtime`、`/settings/runtime/templates` 与 `/settings/workers` 仍保留为可直达的深链接，并自动打开对应子选项卡
  - `/settings/llm` 应提供页面级新手引导卡，说明 `预设 -> 填 key -> 启用 -> 测试 -> 回到对话`
  - `/settings/llm` 的主工作区和关键受阻状态也应镜像当前第一个未完成的 LLM 设置步骤，并给出明确动作
  - `/settings/runtime` 应提供页面级新手引导卡，解释“配置框架 -> 激活 profile -> 验证就绪 -> 继续验证”的顺序，并显示真实状态完成信号
  - `/settings/runtime` 的主工作区以及配置/就绪相关空态也应镜像当前第一个未完成的 Runtime 设置步骤，并给出明确动作
  - `/settings/runtime` 只负责 Runtime 配置与就绪检查；连接示例/模板必须跳转到 `/settings/runtime/templates`，不能继续以内嵌完整模块混在同页
  - `/settings/runtime` 应作为紧凑型工程控制台样板页：
    - 页头保持紧凑，只保留一个主动作：`执行就绪检查`
    - KPI 行仅保留：
      - 可达框架
      - 未配置框架
      - 当前激活 Profile
      - 待处理问题
    - onboarding / bootstrap / 人工确认提示应合并为一份 checklist：
      - 未完成时展开
      - 完成后折叠成一条状态摘要
    - Framework 区改为表格优先，列为：
      - Framework
      - 状态
      - Endpoint
      - API Key
      - 最近检查
      - 操作
	    - 右侧检查栏默认只保留一个紧凑“下一步动作”卡，首屏不再重复堆叠 Runtime/Worker 摘要卡
	    - Runtime 就绪度默认只显示摘要：
      - 错误数
      - 警告数
      - 建议数
      - 当前运行模式
    - 原始 issue code、修复命令、回调 payload、内部诊断默认后移到高级区或抽屉
  - `/settings/runtime` 应把两条 Runtime 接入路径明确展示给新手：
    - `本地模式`：通过当前机器上的本地命令 / 内置 runner 执行，保持 endpoint 为空，且不应再要求填写 API key
    - `Endpoint 模式`：调用远程 Runtime endpoint，继续显示 endpoint，并仅在需要远程鉴权时才展示 API key
  - `/settings/runtime` 首屏应先展示清晰的接入路径选择，再暴露底层维护项：
    - `本地快速配置`：推荐给单机首次使用者，决策最少
    - `激活 profile`：适合已经准备好部署/runtime profile 的环境
    - `自定义框架配置`：适合需要逐框架编辑模式、endpoint、鉴权与本地覆盖项的操作人员
  - `/settings/runtime` 应提供一键“准备纯本地草稿”动作，让新用户无需手动清空 endpoint / 鉴权字段，就能切到本地优先配置
  - `/settings/runtime` 还应提供一键“应用本地快速配置”动作，在就绪检查前一步完成“纯本地草稿 + 当前 runtime 控制项”落盘
  - 低频 Runtime 控件默认应收纳进 `AdvancedSection`，包括：
    - 覆盖式 auto-config / clear settings 等维护动作
    - 严格回退控制项
    - 原始本地命令覆盖项
    - 模型/模型版本级的 endpoint 鉴权路由
  - `/settings/runtime/templates` 应作为 Runtime 连接模板唯一页面：
    - 只承载可复制的环境变量、health-check curl、请求/响应示例
    - 不承载 runtime 就绪状态机、worker 生命周期操作、profile 激活动作
    - Runtime 主页仅保留该页面的入口与简要说明
  - `/settings/workers` 应作为 Worker 运维唯一主页面：
    - 页头只保留一个主动作：`Add Worker`
    - KPI 行聚焦容量与上线状态（`online`、`draining`、`offline`、`pending pairing`）
    - Worker 注册表与行详情抽屉在该页处理
    - Add/Edit/Remove/Enable/Draining 等状态动作仅在该页提供
    - 配对会话（token、setup URL、回调验证、激活）仅在该页提供
    - Runtime 页只保留跳转入口，不再展示 Worker 运维模块或 Worker 容量 KPI

### 3.10 管理域
- `/admin/models/pending`
- 审批队列页应以“待处理请求列表/表格 + 决策抽屉”为核心，其他管理域功能仅保留跳转入口
- `/admin/audit`
- 页面级新手引导卡需解释如何阅读治理记录、区分用户事件与系统事件，并返回其他管理工作线
- `/admin/verification-reports`
- 页面级新手引导卡需强调该页单一主任务：筛选、审查并导出部署验证证据，以支撑发布治理

## 4. 共享 UI 合同
- 路由级页面模块默认采用懒加载，避免未访问工作区阻塞首屏渲染
- `AppShell`：统一导航与页面框架，支持分组导航、当前路由上下文、桌面端折叠与移动端抽屉
- `LeftSidebarShell`：固定高度左侧栏，内部滚动，并支持折叠次级区块与控制台导航分组
- `SessionMenu`：控制台与聊天入口共用的账号胶囊菜单，统一承载设置/退出登录等会话操作
- `StateBlock`：空态/加载/错误/成功统一块
- `AttachmentUploader`：附件可见 + 可删 + 状态展示
- `StepIndicator`：多步骤流程强制使用
- `AdvancedSection`：仅低频或高级控制项默认折叠；当兼容输入成为当前唯一主路径时，不应再额外折叠
- 页面级引导卡应共享一致的“隐藏 / 重新显示”行为，并按路由在本地持久化隐藏状态
- 页面级引导卡还应向统一的“当前页帮助入口”供给内容（右上角固定按钮 + 轻量提示面板），确保隐藏内联卡后仍可随时获得帮助
- 当内联引导仍显示时，默认不要重复整份步骤清单；仅在用户主动展开时才显示详细 checklist
- `DatasetItemBrowser`：用于样本网格/列表、筛选与批量动作的共享浏览组件
- `BulkActionBar`：用于样本级批量 split/status/tag/metadata 动作的共享条
- `SampleReviewWorkbench`：用于标注/复核场景的统一样本工作台布局
- 工程控制台应逐步统一到一套紧凑页面组件：
  - `PageHeader`：紧凑标题 + 一句用途说明 + 唯一主动作
  - `KPIStatRow`：只展示当前页面最关键的 3-5 个指标
  - `FilterToolbar`：统一搜索 / 筛选 / 刷新条
  - `StatusTable`：用于 jobs / models / datasets / audit / reports / workers 的表格型主列表
  - `HealthSummaryPanel`：默认只显示就绪度 / 风险摘要，不直接展开原始诊断
  - `SectionCard`：轻量内容分区容器
  - `InlineAlert`：页内短提示，替代大块提示卡
  - `DetailDrawer`：列表行的次级详情面板
  - `ActionBar`：主次动作分级容器
  - `ConfirmDangerDialog`：删除 / 停用 / 清空 / 撤销类危险操作确认框
- 后台刷新必须感知页面可见性，并且只在存在瞬态状态时运行（例如 `uploading`、`processing`、活动训练任务、活动审核队列）
- 高密度运维列表（例如数据集条目、标注队列）优先使用固定高度内部滚动加 windowed rendering，而不是一次性挂载全部行

## 4.1 控制台页面模板
- 列表页：
  - `PageHeader -> KPIStatRow -> FilterToolbar -> StatusTable -> 可选 DetailDrawer`
  - 需要比较 / 排序 / 筛选的重复行内容，不再优先使用堆叠卡片
- 详情页：
  - `PageHeader -> 上下文摘要 -> Tabs -> 紧凑侧边元信息`
  - 其他重工作流应跳转到专门页面，而不是内嵌
- 设置页：
  - `PageHeader -> HealthSummaryPanel -> 配置表单 -> 高级项折叠区`
  - 诊断细节与维护动作默认后移
- 审计 / 日志页：
  - `PageHeader -> FilterToolbar -> StatusTable -> 原始详情抽屉`
  - 原始 payload 与技术键名默认隐藏，按需展开
- 验证 / 结果页：
  - `PageHeader -> 输入区 -> 结果区 -> 反馈动作区`
  - runtime / 数据 / 训练相关问题仅以摘要 + 正确跳转入口出现

## 5. 页面交互合同

### 5.1 数据集详情页
- 顶部 stepper：`Upload -> Organize -> Version`
- 对话附件通过草稿标签、消息历史或按需托盘可追溯
- 切分/版本是显式动作
- 样本浏览器需支持“浏览 + 分流 + 批量操作”，而不仅是静态详情列表
- 对重复性整理动作需提供样本级批量操作入口
- 同页持续显示标注摘要，让用户在离开数据集前就能看到待审核、返工、已通过等数量
- 提供直达标注工作台的队列快捷入口，并可带上队列、样本或数据集版本上下文
- 版本操作链接应支持把数据集/版本上下文透传到训练任务页与推理验证页
- 首次进入页面可以显示阻塞式加载态，但后台刷新必须保持安静，只在数据集/附件/样本/版本数据真实变化时更新可见内容
- 同时保留手动刷新入口，方便操作人员在需要时显式拉取最新状态

### 5.2 标注工作台
- 顶部 stepper：`Select Item -> Annotate -> Review`
- 样本复核台需在一个工作面内对齐样本预览、标注数据、元数据与最近审核上下文
- 样本侧栏支持 `all`、`needs_work`、`in_review`、`rejected`、`approved` 等聚焦队列，不应在后台无变化时丢失当前选择
- 页面首屏应先呈现当前样本、主要标注动作与复核入口，不应要求用户先阅读会话统计、快捷键、返回点等高阶辅助信息
- 复核会话历史、低置信分流提示和其他熟练用户工具应继续保留，但默认作为次级能力呈现
- 最近一次审核详情卡在返工过程中持续可见，帮助标注人员理解被拒原因
- 当用户明确将 `rejected` 条目移回 `in_progress` 时，工作台应继续保留当前样本，并自动切到 `needs_work` 队列，避免返工时丢失上下文
- 数据集详情页可通过深链接预选当前队列、样本，并在可用时保留数据集版本上下文
- 首次进入页面可以显示阻塞式加载态，但后台刷新不得在服务端数据未变化时反复重置当前样本或画布
- 同时保留手动刷新入口，方便标注人员按需拉取最新审核/预标注状态

### 5.3 训练任务创建页
- 顶部 stepper：`Task -> Data -> Params -> Review`
- 训练提交前，数据集选择必须绑定明确的数据集版本快照，不能直接依赖隐式“最新数据集状态”
- 页面需持续展示当前所选版本的启动准备摘要：数据集状态、切分摘要、标注覆盖率，以及是否存在可训练的 `train` 切分
- 当所选版本 `annotation_coverage <= 0` 或 `train` 切分为空时，启动动作必须保持禁用
- 高级参数默认折叠

### 5.4 推理验证页
- 复用上传组件与状态块
- runtime 区仅保留只读摘要（reachable/unreachable/not configured 计数）
- 完整 runtime 诊断与配置动作必须保留在 `/settings/runtime`，通过明确跳转进入
- 输出面板包含：
  - 模型元信息
  - 原始输出
  - 归一化输出
- 一键回流数据集
- 首次进入页面可以显示阻塞式加载态，但后台刷新必须保持安静，只在模型版本/数据集/附件/运行记录真实变化时更新可见内容

### 5.5 模型概览页
- `/models/explore`、`/models/my-models`、`/models/versions` 统一使用共享 overview 布局（`hero -> signals -> 主列表 + 侧边动作`）
- 主库存信息放在主栏，创建或跟进行动保持在侧栏持续可见
- 加载、空态、错误、成功统一复用 `StateBlock` 语义，与数据集页、训练页保持一致

### 5.6 设置总入口
- `/settings` 是控制台左侧导航中的唯一设置入口
- 子选项卡停留在页面内部，用户可在 `LLM`、`Runtime`、`Runtime Templates` 与 `Workers` 之间切换，而不需要回到全局导航重复寻找
- 当前选中的子选项卡必须在页头中保持清晰可见，同时深链接仍可直接打开特定子页
- `Account` 子页应提供同时适配普通用户与管理员的新手引导步骤，并给出直达后续设置子页的入口
- 设置入口卡片应展示紧凑的 Runtime 快速上手快照（`进度`、`下一步动作`、`纯本地已启用或 endpoint 数量`），让用户在进入 Runtime 子页前就能理解当前配置状态

### 5.7 LLM 设置子选项卡
- 统一使用 shared overview 布局，包含已保存配置摘要、可编辑表单，以及动作/排查侧边区
- 编辑过程中仍持续显示掩码 key 状态，让用户知道保存/测试是否会复用已存储密钥
- 保存、测试、重新加载、清空、预设动作保持靠近表单，同时不隐藏连接建议
- 子页引导卡需按初学者顺序给出配置步骤，并根据已保存配置与连接测试状态计算完成度

### 5.8 Runtime 设置子选项卡
- 支持全框架与单框架连通性检查
- 该子页只负责 Runtime 配置与就绪检查；连接模板应跳转到 `/settings/runtime/templates`
- 通过统一 shared overview 布局，将诊断与执行摘要保持在同一页面层级，避免混入独立模板模块
- 执行摘要还应直接显示“近期已完成训练任务中按框架采样得到的最新指标键”，避免用户必须跳转到训练详情页逐个查看
- 子页引导卡需按初学者顺序给出关键动作，并从 runtime/profile/readiness 真实记录计算完成状态
- 每个框架卡都应提供明确的连接模式切换（`本地模式` / `Endpoint 模式`），不要把两种心智模型混在同一组表单里
- 每个框架卡都应提供模型感知配置：
  - 默认模型 + 可选默认模型版本下拉，来源需同时包含基座模型与可用的已发布/已注册模型
  - Endpoint 模式除 framework 级 API key 外，还应支持模型级 key 绑定（`model:*`、`model_version:*`）
- 当框架卡处于 `本地模式` 时：
  - endpoint 输入应隐藏或明显弱化
  - API key 输入应隐藏，因为本地执行不需要它
  - 本地命令 / 本地模型路径说明应成为主解释内容
- 当框架卡处于 `Endpoint 模式` 时：
  - endpoint 输入保持主入口
  - API key 继续保留为“远程端点可选鉴权”字段
- 在分布式 worker / 高级控制之前，页面应先提供一条紧凑的单机快速配置路径，让零基础用户先完成可用的本地 Runtime 设置
- 该快速路径应明确按顺序提供动作：`准备纯本地草稿 -> 应用本地快速配置 -> 运行就绪检查`
- 引导完成度计算应将“已保存的纯本地配置”视为有效的“框架已配置基线”，即使未配置外部 endpoint 也不应卡在第一步
- 快速路径区块应展示步骤级进度/完成信号，并在任意时刻只突出一个“建议下一步”；全部完成后提供直达推理验证入口

### 5.8.1 Runtime Templates 设置子选项卡
- 作为 Runtime 连接模板的独立页面
- 提供可复制的环境变量名、health-check curl 与请求/响应示例
- 不承载 Runtime profile 激活、就绪状态机与 Worker 生命周期操作
- 通过入口返回 Runtime 设置页完成真实配置与就绪检查

### 5.9 Workers 设置子选项卡
- Worker 注册表应采用表格优先，行级详情在抽屉中查看
- Worker 配对流程只在本页承载：setup URL、token、回调验证、激活
- Worker 变更动作只在本页承载：add/edit/remove、enable/disable、draining/resume、reconfigure
- 页面保持单一主动作（`Add Worker`），其他动作按 secondary/tertiary 降级

### 5.10 管理员验收报告页
- 支持按状态、base_url、关键词筛选
- 支持日期区间筛选与排序（最新优先/最早优先/失败优先）
- 支持快捷日期筛选（近 7 天 / 近 30 天 / 清空）
- 报告检查项明细折叠查看，降低长列表噪音
- 支持分页浏览与“按筛选结果导出 JSON”
- 默认排序优先展示失败报告，便于治理排障
- 用于发布交付前治理校验

## 6. 响应式基线
- 移动端：单列堆叠
- 桌面端：侧边导航 + 内容区
- stepper 与附件状态在各断点保持可读

## 7. 阶段边界
- Phase 1：骨架页 + mock API
- Phase 2：最小在线标注与审核闭环
- Phase 3+：真实框架适配器与训练执行器
- 视觉数据闭环演进轨道（参考 `docs/visual-data-loop-evolution.zh-CN.md`）以增量方式升级数据浏览/复核/版本流程，不改变 chat-first IA 主轴
